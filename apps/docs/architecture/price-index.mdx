<Note>
**Role downgraded by the voucher decision.** Originally the index was *the product*. After the
`/grill-me` session, the [redeemable voucher](/architecture/instrument) anchors price by arbitrage,
so for the MVP the index is a **reference / UX layer + the underwriter's risk gauge** — **not** the
thing settlement depends on. It only becomes trust-critical again if a *non-redeemable derivative*
layer is added later. Build it transparent and reproducible regardless; just don't bet settlement on
it in v0.
</Note>

## Design goals

<CardGroup cols={2}>
<Card title="Transparent & reproducible" icon="eye">
Anyone can recompute today's value from published inputs and a published formula. This is OTE's
differentiator vs Silicon Data's *proprietary* index.
</Card>
<Card title="Manipulation-resistant" icon="shield-halved">
No single provider, model, or moment can swing the number. Multi-source, time-averaged, outlier-
trimmed.
</Card>
<Card title="Capability-stable" icon="ruler">
A SIT-style variant holds *quality constant* so the series tracks "cost of a capability," not "we
switched to a smaller model."
</Card>
<Card title="Multi-dimensional → scalar" icon="compress">
Collapses the `model × provider × {in,out} × time` vector into defensible numbers via an explicit,
versioned methodology.
</Card>
</CardGroup>

## Methodology (v0 proposal)

<Steps>
<Step title="Ingest">
Poll the **OpenRouter models/pricing API** on a fixed cadence (e.g. every 5 min). Record, per
`(model, provider, endpoint)`: input $/M, output $/M, timestamp. **Add ≥1 second source** (another
aggregator or direct provider rate cards) before any *settlement* use, to avoid single-venue
dependence.
</Step>
<Step title="Per-model price">
For each model, combine the cheapest **k** healthy providers (e.g. k=3) into a blended rate, with a
fixed **input:output ratio** (e.g. 1:3, documented) so prompt/completion collapse into one $/M
number. Trim outliers; drop stale/unavailable endpoints.
</Step>
<Step title="Index series">
Publish three layers:
1. **Per-model index** — `OTE-<model>` (e.g. `OTE-GPT5`).
2. **Class index** — baskets like `OTE-FRONTIER`, `OTE-CHEAP-BULK`, weighted by usage or equal-weight.
3. **SIT-normalized index** — the cheapest model *meeting a fixed capability bar* (MMLU/HumanEval/
   GSM8K thresholds), so the series measures cost-of-capability. See [Analogues](/research/analogues).
</Step>
<Step title="Settlement value">
For any cash-settled instrument, settle to a **time-averaged window** (e.g. TWAP over the final
24–72h), **not a single tick** — the standard manipulation defense. Publish a **dispute window**
before finalization.
</Step>
</Steps>

## Manipulation-resistance checklist

<Warning>
Because the underlying is administratively set and sourced from few venues, assume an adversary can
influence *a* provider's listed price. Defenses:

- **Multi-source** — ≥2 independent aggregators / direct rate cards; never settle off one venue.
- **Time-averaging (TWAP)** — settle on a multi-hour/day window, not a spot tick.
- **Outlier trimming** — drop the top/bottom x% of provider quotes before blending.
- **k-of-N provider blend** — require several providers to move before the index moves.
- **Capability gate** — a suspiciously cheap "new provider" that fails the SIT benchmark bar doesn't
  enter the normalized index.
- **Published methodology + versioning** — changes are announced, diffed, and timestamped.
- **Dispute window** — settlement values are provisional for N hours before finalization.
</Warning>

## The underwriter's reset study (Phase 0 deliverable)

<Info>
The Phase‑0 measurement was **reframed** in the grill: as the [underwriter](/architecture/instrument),
OTE does **not** need day-trading volatility — it needs to know how safely it can sell a below-spot
voucher. So the study measures:

- **Decline forecastability** — how predictable is the secular downtrend per model/basket? (sets the
  size of the discount OTE can offer).
- **Upward-reset frequency & magnitude** — how often, and how hard, does a price *reset up* (frontier
  launch, provider hike, decline stall)? (sets **reserve sizing** and **position limits**).
- **Reprice cadence** — how often any tracked provider changes a listed rate (sets index refresh +
  redemption-pricing latency).

**Decision rule:** if upward resets are rare and small, OTE can underwrite aggressively with a fat
discount. If they're frequent/large, the discount shrinks (or OTE waits for provider lay-off before
being the house). Either way it's a **cheap, early finding** — before a single voucher is minted.
</Info>

## Data model (sketch)

```ts
// packages/index-core — the price record and published index point
type PriceObservation = {
  model: string;          // "openai/gpt-5"
  provider: string;       // "deepinfra"
  inputUsdPerM: number;
  outputUsdPerM: number;
  observedAt: string;     // ISO8601
  available: boolean;
};

type IndexPoint = {
  symbol: string;         // "OTE-GPT5" | "OTE-FRONTIER" | "OTE-SIT"
  value: number;          // blended USD / M tokens (input:output ratio fixed)
  method: string;         // semver of the methodology, e.g. "ote-idx@0.1.0"
  window: "spot" | "twap-24h" | "twap-72h";
  sourcesUsed: number;    // provider count after trimming
  computedAt: string;
};
```

<Tip>
Keep the methodology in a **versioned package** (`@ote/index-core`) so the published `method` field
on every `IndexPoint` is auditable. A trusted index is a *reproducible* index.
</Tip>
